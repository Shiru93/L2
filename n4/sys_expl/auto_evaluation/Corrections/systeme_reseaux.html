Q1 Comment écrire en C, compiler et exécuter un Hello, World! sur ma propre machine ? +

1. Créer un fichier hello.c avec un éditeur de texte :

#include <stdio.h> int main(void) { printf("Hello, World!\n"); return 0; }

2. Compiler avec GCC :

gcc -Wall -Wextra -o hello hello.c

3. Exécuter :

./hello

Prérequis Avoir GCC installé. Sur Ubuntu/Debian : sudo apt install gcc. Sur macOS : xcode-select --install.

Q2 Quel éditeur de texte j'utilise pour écrire des programmes en C ? +

Pas de mauvaise réponse L'important est d'en connaître un.

  • Sur sa propre machine : VS Code (recommandé), CLion, Eclipse, Vim/Neovim, Emacs…
  • En salle TP : VS Code, nano (simple), gedit (graphique)

VS Code avec l'extension C/C++ offre la coloration syntaxique, l'autocomplétion et l'intégration du débogueur.

Q3 Quels drapeaux de compilation sont indispensables ? Fortement conseillés ? +

Indispensables dans ce cours :

gcc -std=c99 -o mon_prog mon_prog.c
  • -std=c99 (ou -std=c11) : impose le standard C99/C11, nécessaire pour les types int32_t etc.

Fortement conseillés :

gcc -Wall -Wextra -g -std=c99 -o mon_prog mon_prog.c
  • -Wall : active la plupart des avertissements utiles (variables non utilisées, retour manquant…)
  • -Wextra : avertissements supplémentaires (comparaisons, paramètres inutilisés…)
  • -g : inclut les infos de débogage, indispensable pour utiliser GDB ou Valgrind efficacement
  • -Werror : (optionnel) traite les avertissements comme des erreurs — force à corriger tout

Règle d'or Compiler toujours avec -Wall -Wextra -g. Les warnings sont presque toujours des bugs.

Q4 Comment utiliser Valgrind ? À quoi cela sert-il ? +

Rôle : Valgrind est un outil de détection dynamique d'erreurs mémoire. Il permet de trouver :

  • Les fuites mémoire (malloc sans free)
  • Les accès hors limites (dépassement de tableau)
  • La lecture de mémoire non initialisée
  • Le double free

Utilisation :

# D'abord compiler avec -g gcc -Wall -Wextra -g -o mon_prog mon_prog.c # Puis lancer avec valgrind valgrind --leak-check=full ./mon_prog [arguments]

--leak-check=full affiche les détails de chaque fuite mémoire avec la pile d'appels.

Attention Valgrind ralentit l'exécution d'un facteur 10 à 50×, mais c'est normal.

Q5 Qu'est-ce qu'un bit et un octet (byte) ? +
  • Bit : unité d'information la plus petite. Ne peut valoir que 0 ou 1.
  • Octet (byte) : groupe de 8 bits. C'est l'unité d'adressage standard en mémoire.

Exemple : l'octet 01000001 représente la valeur 65, qui correspond au caractère 'A' en ASCII.

Q6 Que signifie « la machine est byte addressable » ? +

Cela signifie que chaque octet en mémoire possède une adresse unique. Le plus petit élément adressable est l'octet (et non le bit, ni un mot de 4 octets).

Conséquence : un pointeur char * permet d'accéder à chaque octet individuellement. Deux pointeurs de types différents pointant vers la même adresse pointent vers le même premier octet.

Q7 Quelle est la différence entre les entiers signés et non signés ? +
  • Signés (signed) : peuvent représenter des valeurs positives ET négatives. Sur n bits, plage : −2^(n−1) à +2^(n−1)−1. Ex: int 32 bits → −2 147 483 648 à +2 147 483 647.
  • Non signés (unsigned) : valeurs positives uniquement. Sur n bits, plage : 0 à 2^n−1. Ex: unsigned int 32 bits → 0 à 4 294 967 295.

En C, les entiers sont signés par défaut (sauf char qui dépend de la machine).

La représentation utilisée pour les signés est le complément à 2.

Q8 Quelle est la taille d'un char ? Est-il signé ou non signé ? +
  • Taille : toujours 1 octet (8 bits) — c'est garanti par la norme C.
  • Signé ou non : dépend de la machine et du compilateur. Sur x86/x86-64 avec GCC, char est signé par défaut. Sur certaines architectures ARM, il est non signé.

Pour lever l'ambiguïté, utiliser explicitement signed char ou unsigned char.

Q9 Quelle est la taille d'un int ? Est-il signé ou non signé ? +
  • Taille : généralement 4 octets (32 bits) sur les architectures modernes 32 et 64 bits (x86-64, ARM64). La norme garantit seulement ≥ 16 bits.
  • Signé : int est signé par défaut. Pour non signé : unsigned int.

Pour une taille garantie, utiliser les types de <stdint.h> : int32_t, uint32_t…

Q10 Combien de valeurs peut-on représenter sur 8, 16, 32 bits ? +
  • 8 bits : 2⁸ = 256 valeurs (0 à 255 non signé, −128 à 127 signé)
  • 16 bits : 2¹⁶ = 65 536 valeurs (0 à 65535 non signé)
  • 32 bits : 2³² = 4 294 967 296 ≈ 4 milliards de valeurs
  • 64 bits : 2⁶⁴ ≈ 1,8 × 10¹⁹ valeurs
Q11 Est-ce que je connais toutes les puissances de 2 jusqu'à 2¹⁰ ? +
2⁰ = 1 2¹ = 2 2² = 4 2³ = 8 2⁴ = 16 2⁵ = 32 2⁶ = 64 2⁷ = 128 2⁸ = 256 2⁹ = 512 2¹⁰ = 1024 ≈ 10³

Astuce mnémotechnique 2¹⁰ = 1024 ≈ 1 kilo. C'est la base pour estimer les grandes puissances de 2.

Q12 Comment donner une valeur décimale approchée d'une grande puissance de 2 ? +

On utilise l'approximation 2¹⁰ ≈ 10³ :

2³² = (2¹⁰)³ × 2² ≈ (10³)³ × 4 = 4 × 10⁹ ≈ 4 milliards 2⁶⁴ = (2¹⁰)⁶ × 2⁴ ≈ (10³)⁶ × 16 ≈ 1,6 × 10¹⁹ 2²⁰ = (2¹⁰)² ≈ (10³)² = 10⁶ = 1 million (1 méga)

Mémo : 2¹⁰ ≈ kilo (10³), 2²⁰ ≈ méga (10⁶), 2³⁰ ≈ giga (10⁹).

Q13 Comment convertir un entier décimal en binaire ? en hexadécimal ? +

Décimal → Binaire : divisions euclidiennes successives par 2, lire les restes de bas en haut.

13 ÷ 2 = 6 reste 1 6 ÷ 2 = 3 reste 0 3 ÷ 2 = 1 reste 1 1 ÷ 2 = 0 reste 1 → 13 en binaire = 1101

Décimal → Hexadécimal : même principe mais divisions par 16. Les restes 10–15 s'écrivent A–F.

255 ÷ 16 = 15 reste 15 (= F) 15 ÷ 16 = 0 reste 15 (= F) → 255 en hexa = 0xFF
Q14 Comment passer du binaire à l'hexadécimal et réciproquement ? +

Binaire → Hexadécimal : regrouper les bits par 4 en partant de la droite. Chaque groupe de 4 bits donne un chiffre hexadécimal.

10110101 = 1011 | 0101 = B | 5 = 0xB5

Hexadécimal → Binaire : chaque chiffre hexa se convertit en 4 bits.

0x3F = 0011 1111

Table à mémoriser

0=0000 1=0001 2=0010 3=0011 4=0100 5=0101 6=0110 7=0111 8=1000 9=1001 A=1010 B=1011 C=1100 D=1101 E=1110 F=1111
Q15 Quels sont les types entiers de taille et signitude fixes en C99 ? +

Définis dans <stdint.h> (C99) :

// Signés int8_t int16_t int32_t int64_t // Non signés uint8_t uint16_t uint32_t uint64_t // Plus rapides sur la machine (≥ n bits) int_fast8_t int_fast32_t ... // Plus petits en mémoire (≥ n bits) int_least8_t int_least32_t ... // Pour stocker un pointeur intptr_t uintptr_t // Taille maximale disponible intmax_t uintmax_t

Bonne pratique Préférer int32_t à int quand la taille exacte compte (protocoles réseau, fichiers binaires…).

Q16 Qu'est-ce que le boutisme (endianness) ? Différence little-endian / big-endian ? +

Le boutisme définit l'ordre dans lequel les octets d'un entier multi-octet sont stockés en mémoire.

Exemple : l'entier 32 bits 0x12345678 stocké à l'adresse 0x1000 :

Adresse : 0x1000 0x1001 0x1002 0x1003 Little-endian (x86, x86-64) : 0x78 0x56 0x34 0x12 ↑ octet de poids FAIBLE en premier Big-endian (réseau, certains ARM) : 0x12 0x34 0x56 0x78 ↑ octet de poids FORT en premier

Importance Crucial lors de la communication réseau (réseau = big-endian) ou de la lecture de fichiers binaires. Les fonctions htons(), ntohl() etc. servent à convertir.

Q17 Révisions sur les pointeurs : opérateurs & et * ? +
  • &x : opérateur adresse de. Retourne l'adresse mémoire de la variable x.
  • *p : opérateur de déréférencement. Accède à la valeur pointée par le pointeur p.
int x = 42; int *p = &x; // p contient l'adresse de x *p = 100; // modifie x via p → x vaut maintenant 100 printf("%d\n", x); // affiche 100

* dans une déclaration signifie "pointeur vers". * dans une expression signifie "déréférencer".

Noyau, Processus et fork()
Système d'exploitation, appels système, création de processus
Q1 Quels sont les rôles principaux du noyau d'un système d'exploitation ? +

Le noyau (kernel) est le cœur du système d'exploitation. Ses rôles principaux :

  • Gestion des processus : création, ordonnancement, terminaison
  • Gestion de la mémoire : allocation, protection, mémoire virtuelle
  • Gestion des entrées/sorties : pilotes de périphériques, abstraction matérielle
  • Système de fichiers : organisation, accès et protection des fichiers
  • Communication inter-processus (IPC) : signaux, pipes, sockets…
  • Sécurité et protection : espace utilisateur vs espace noyau, permissions

Le noyau s'exécute en mode privilégié. Les programmes utilisateurs ne peuvent accéder aux ressources matérielles qu'à travers lui, via les appels système.

Q2 Qu'est-ce qu'un appel système ? Exemples ? +

Un appel système (syscall) est une interface par laquelle un programme en mode utilisateur demande un service au noyau. C'est le seul moyen légal d'accéder aux ressources matérielles.

Mécanisme : le programme déclenche une interruption logicielle → passage en mode noyau → exécution → retour en mode utilisateur.

Exemples :

  • Processus : fork(), execve(), exit(), wait(), getpid()
  • Fichiers : open(), read(), write(), close()
  • Mémoire : mmap(), brk()
  • Signaux : kill(), sigaction()
  • Réseau : socket(), bind(), connect()
Q3 Que contient la section 2 du manuel ? Et la section 3 ? +
  • Section 2 : les appels système (interfaces directes avec le noyau). Ex : fork, read, write…
  • Section 3 : les fonctions de la bibliothèque C standard (libc). Ex : printf, malloc, strlen…
man 2 fork # appel système fork man 3 printf # fonction de la bibliothèque standard man 2 open # appel système open man 3 fopen # fonction stdio (enveloppe de open)

Autres sections utiles : 1 = commandes shell, 4 = fichiers spéciaux, 5 = formats de fichiers, 7 = aperçus.

Q4 Qu'est-ce qu'un processus ? Qu'est-ce qui l'identifie ? Quels sont ses attributs ? +

Un processus est un programme en cours d'exécution. C'est une instance d'un programme avec son propre espace mémoire, ses ressources et son état d'exécution.

Identifiant : le PID (Process IDentifier), un entier unique attribué par le noyau.

Attributs d'un processus :

  • PID : identifiant du processus
  • PPID : PID du processus parent
  • UID / GID : identité de l'utilisateur/groupe propriétaire
  • Espace d'adressage (code, pile, tas, données)
  • Table des descripteurs de fichiers ouverts
  • Répertoire courant
  • Variables d'environnement
  • État (running, sleeping, zombie, stopped…)
  • Priorité d'ordonnancement
Q5 Comment chercher l'identifiant d'un processus ? Qu'est-ce que ça permet ? +
ps aux # liste tous les processus ps aux | grep nom # filtrer par nom pgrep nom # retourne les PID par nom pidof nom # PID d'un exécutable par son nom top / htop # vue dynamique

Une fois le PID connu, on peut :

  • Envoyer un signal : kill -SIGTERM 1234 ou kill -9 1234
  • Attacher un débogueur : gdb -p 1234
  • Inspecter la mémoire : /proc/1234/maps
  • Voir les fichiers ouverts : lsof -p 1234
Q6 Instruction switch en C : syntaxe, cas, défaut, break ? +
switch (expression) { case valeur1: // instructions break; // IMPORTANT : évite le "fall-through" case valeur2: case valeur3: // les deux cas font la même chose // instructions break; default: // cas par défaut (optionnel) // instructions break; // bonne pratique de l'inclure }

break : sort du switch. Sans break, l'exécution continue dans le case suivant (fall-through), ce qui est rarement voulu.

Piège classique Oublier un break provoque un bug silencieux où plusieurs cases s'exécutent. Compiler avec -Wimplicit-fallthrough pour être averti.

Q7 Quel appel système crée un processus ? Quelle valeur retourne-t-il ? +

fork() crée un processus fils qui est une copie exacte du père (espace mémoire, descripteurs, etc.).

#include <unistd.h> pid_t pid = fork(); if (pid == -1) { // ERREUR (fork a échoué) perror("fork"); } else if (pid == 0) { // CODE DU FILS (fork retourne 0 au fils) printf("Je suis le fils, mon PID = %d\n", getpid()); } else { // CODE DU PÈRE (fork retourne le PID du fils au père) printf("J'ai créé le fils avec PID = %d\n", pid); }

Valeurs de retour : 0 → au fils | PID du fils → au père | −1 → erreur

Q8 Pourquoi parle-t-on d'arborescence des processus ? +

Chaque processus a un parent (sauf le tout premier). Quand un processus crée un fils via fork(), cela forme une relation parent-enfant. L'ensemble forme une arborescence.

  • Au démarrage du système, le noyau crée le processus init (PID 1) ou systemd.
  • Tous les autres processus sont des descendants de PID 1.
systemd (PID 1) ├── sshd (PID 234) │ └── bash (PID 567) │ └── gcc (PID 890) ├── Xorg (PID 345) └── ...

pstree permet de visualiser l'arborescence complète.

Q9 Que sont les arguments d'un processus ? Comment y accéder ? +

Les arguments sont les paramètres passés au programme lors de son lancement (ex : ./prog arg1 arg2).

int main(int argc, char *argv[]) { // argc = nombre d'arguments (y compris le nom du prog) // argv[0] = nom du programme // argv[1] = premier argument // argv[argc-1] = dernier argument // argv[argc] = NULL (sentinelle) for (int i = 0; i < argc; i++) { printf("argv[%d] = %s\n", i, argv[i]); } return 0; }

Exemple : ./mon_prog hello 42 → argc=3, argv={"./mon_prog", "hello", "42", NULL}

Q10 Qu'est-ce que l'environnement d'un processus ? Comment y accéder en C ? +

L'environnement est un ensemble de variables de la forme NOM=valeur transmises au processus. Ex : PATH, HOME, USER…

#include <stdlib.h> // Méthode 1 : getenv (la plus simple) char *path = getenv("PATH"); if (path != NULL) printf("PATH = %s\n", path); // Méthode 2 : via le 3ème paramètre de main int main(int argc, char *argv[], char *envp[]) { for (int i = 0; envp[i] != NULL; i++) printf("%s\n", envp[i]); } // Méthode 3 : variable globale extern extern char **environ; for (char **e = environ; *e != NULL; e++) printf("%s\n", *e);
Q11 Comment un processus peut-il connaître son PID ? Et celui de son parent ? +
#include <unistd.h> pid_t mon_pid = getpid(); // PID du processus courant pid_t pid_parent = getppid(); // PID du processus parent printf("Mon PID : %d, PID parent : %d\n", mon_pid, pid_parent);

Ces deux fonctions ne retournent jamais d'erreur.

Signaux, wait() et exec()
Gestion d'erreurs, signaux Unix, attente de processus, remplacement d'image
Q1 À quoi servent perror et la variable globale errno ? +
  • errno : variable globale (dans <errno.h>) qui contient le code d'erreur du dernier appel système échoué. Valeur : entier positif (ex : ENOENT=2, EACCES=13…). N'est significative que si l'appel système a retourné −1.
  • perror(msg) : affiche sur stderr le message msg suivi de : et du message d'erreur correspondant à errno.
int fd = open("fichier.txt", O_RDONLY); if (fd == -1) { perror("Erreur open"); // Affiche par ex: "Erreur open: No such file or directory" exit(1); }

Règle Toujours vérifier la valeur de retour des appels système et afficher l'erreur avec perror.

Q2 Pourquoi utiliser des arguments pointeurs dans les fonctions ? +

En C, les arguments sont passés par valeur (copie). Un pointeur en argument permet :

  • Modifier la variable de l'appelant (passage par référence simulé) :
void incrementer(int *x) { (*x)++; } int n = 5; incrementer(&n); // n vaut maintenant 6
  • Retourner plusieurs valeurs via plusieurs pointeurs en argument
  • Efficacité : passer l'adresse d'une grande structure plutôt que de la copier

Exemple concret : scanf("%d", &x) utilise un pointeur pour écrire le résultat dans x.

Q3 Quel mot-clé indique qu'un argument pointeur n'est utilisé que pour lire ? +

Le mot-clé const.

// Correct : le pointeur est en lecture seule void afficher(const char *s) { printf("%s\n", s); // s[0] = 'X'; // ERREUR de compilation : lecture seule } // Comparaison : void modifier(char *s) { s[0] = 'X'; } // peut écrire

Bonnes pratiques

  • Met en évidence l'intention (documentation du code)
  • Permet le compilateur de détecter les modifications accidentelles
  • Autorise l'appel avec des chaînes littérales (const char*)
Q4 Qui peut envoyer un signal à un processus ? +
  • Le processus lui-même (ex : raise(SIGTERM))
  • Un autre processus du même utilisateur (via kill())
  • Le superutilisateur (root) : peut envoyer à n'importe quel processus
  • Le noyau : signaux générés par des événements matériels (SIGSEGV = accès mémoire invalide, SIGFPE = division par zéro, SIGCHLD = mort d'un enfant…)
  • Le terminal : Ctrl+C génère SIGINT, Ctrl+Z génère SIGTSTP, Ctrl+\ génère SIGQUIT
Q5 Quelles sont les différentes actions que peut entraîner la réception d'un signal ? +
  • Terminer le processus (action par défaut de nombreux signaux : SIGTERM, SIGKILL…)
  • Ignorer le signal (avec SIG_IGN, sauf SIGKILL et SIGSTOP qui ne peuvent être ignorés)
  • Exécuter un gestionnaire personnalisé (handler) défini par le programmeur avec signal() ou sigaction()
  • Suspendre le processus (SIGSTOP, SIGTSTP)
  • Reprendre un processus suspendu (SIGCONT)
  • Générer un core dump (SIGQUIT, SIGSEGV, etc.)
Q6 Quel est le prototype de la fonction kill ? +
#include <signal.h> int kill(pid_t pid, int sig);

Paramètres :

  • pid : PID du processus destinataire (ou groupe si négatif)
  • sig : signal à envoyer (ex : SIGTERM, SIGKILL, SIGUSR1…)

Retour : 0 si succès, −1 si erreur (errno mis à jour)

kill(pid_fils, SIGTERM); // demande poliment la terminaison kill(pid_fils, SIGKILL); // terminaison forcée (ne peut être bloquée) kill(pid_fils, 0); // vérifie si le processus existe (sans envoyer de signal)
Q7 Quel appel système permet d'attendre la mort d'un enfant ? Comment l'utiliser ? +
#include <sys/wait.h> pid_t waitpid(pid_t pid, int *status, int options);
  • pid : PID de l'enfant attendu, ou −1 pour attendre n'importe quel enfant
  • status : pointeur vers un entier qui recevra l'état de terminaison
  • options : 0 pour bloquer, WNOHANG pour ne pas bloquer
  • Retour : PID de l'enfant terminé, ou −1 si erreur
int status; pid_t pid_mort = waitpid(-1, &status, 0); // attend un enfant quelconque if (WIFEXITED(status)) // s'est terminé normalement ? printf("Code de sortie: %d\n", WEXITSTATUS(status)); if (WIFSIGNALED(status)) // tué par un signal ? printf("Signal: %d\n", WTERMSIG(status));

wait(&status) est équivalent à waitpid(-1, &status, 0).

Q8 Comment exécuter un nouveau programme dans un processus ? Qu'arrive-t-il à l'ancien ? +

La famille de fonctions exec() remplace l'image mémoire du processus courant par un nouveau programme.

#include <unistd.h> execl("/bin/ls", "ls", "-l", NULL); // Si execl réussit, la ligne suivante n'est JAMAIS exécutée perror("execl a échoué");

Ce qui est conservé : PID, PPID, descripteurs de fichiers (sauf ceux marqués FD_CLOEXEC), variables d'environnement…

Ce qui est remplacé : code, données, pile, tas — tout le programme précédent disparaît.

En cas de succès, exec ne retourne jamais. Si ça retourne, c'est une erreur.

Q9 Qu'est-ce que le PATH ? Différence entre execlp et execl ? +

PATH : variable d'environnement contenant une liste de répertoires séparés par : où le shell (et les fonctions execlp/execvp) cherche les exécutables.

echo $PATH # /usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin
  • execl : chemin absolu ou relatif requis. Ex : execl("/bin/ls", "ls", NULL)
  • execlp : cherche l'exécutable dans le PATH. Ex : execlp("ls", "ls", NULL)

Même logique pour execv vs execvp.

Q10 Quelle est la différence entre execv et execl ? +
  • execl : arguments listés un par un dans la fonction (liste variable), terminée par NULL.
  • execv : arguments passés dans un tableau (char *argv[]), terminé par NULL.
// execl : arguments individuels execl("/bin/ls", "ls", "-l", "-a", NULL); // execv : tableau d'arguments char *args[] = {"ls", "-l", "-a", NULL}; execv("/bin/ls", args);

execv est plus pratique quand le nombre d'arguments est variable ou construit dynamiquement.

Q11 Pour quelles raisons un appel exec pourrait-il échouer ? +
  • Le fichier n'existe pas (ENOENT)
  • Pas les droits d'exécution (EACCES)
  • Format d'exécutable invalide ou architecture incompatible (ENOEXEC)
  • Trop d'arguments ou d'environnement (E2BIG)
  • Manque de mémoire (ENOMEM)
  • Le fichier est un répertoire
  • Chemin trop long (ENAMETOOLONG)
Q12 Dans quel cas un processus peut-il être un zombie ? +

Un processus zombie est un processus qui s'est terminé mais dont le père n'a pas encore appelé wait().

Le processus zombie ne consomme plus de CPU ni de mémoire significative, mais son entrée dans la table des processus persiste pour conserver son code de sortie jusqu'à ce que le père le récupère.

// Pour éviter les zombies : // 1. Le père appelle wait() après fork() // 2. Ignorer SIGCHLD : signal(SIGCHLD, SIG_IGN); // 3. Gestionnaire de SIGCHLD qui appelle waitpid()

Problème Si le père meurt sans avoir appelé wait(), l'enfant zombie est "adopté" par init (PID 1) qui appellera wait() à sa place.

void*, Pointeurs de fonctions et Threads
Généricité en C, callbacks, programmation multi-thread avec pthreads
Q1 Qu'est-ce qu'un pointeur void* ? 3 exemples dans la bibliothèque standard ? +

void * est un pointeur générique : il peut pointer vers n'importe quel type, sans connaître le type à la compilation. C'est le mécanisme de généricité du C.

3 exemples dans la bibliothèque standard :

  • void *malloc(size_t size) — alloue de la mémoire, retourne un pointeur générique
  • void *memcpy(void *dest, const void *src, size_t n) — copie n octets
  • void qsort(void *base, size_t nmemb, size_t size, int (*compar)(const void*, const void*)) — tri générique
  • (bonus) void *pthread_create — prend un void * en argument de thread
Q2 Dans quels cas utiliser un pointeur void* ? +
  • Quand on veut écrire une fonction générique qui fonctionne avec n'importe quel type
  • Pour les fonctions de la bibliothèque standard (malloc, qsort, etc.)
  • Pour passer des données à un thread (pthread_create prend un void *)
  • Pour stocker des pointeurs de types différents dans la même structure
  • Quand l'interface doit être indépendante du type concret (abstraction)
Q3 Que ne peut-on PAS faire avec un void* ? +
  • Déréférencer directement : *p est interdit car la taille du type pointé est inconnue
  • Arithmétique de pointeur : p + 1 est interdit (la taille d'un "void" est inconnue)
  • Accès à des champs : p->champ est interdit

Il faut d'abord caster vers le type concret :

void *p = malloc(sizeof(int)); *(int *)p = 42; // cast puis déréférencement : OK int *pi = (int *)p; // conversion explicite : OK pi[0] = 42; // maintenant on peut
Q4 Dans quels cas écrire explicitement un cast entre pointeurs ? +

En C (pas C++) :

  • void * ↔ tout autre pointeur : implicite (pas besoin de cast)
  • Pointeur vers type A → pointeur vers type B (différents) : cast explicite obligatoire
int *p = malloc(sizeof(int)); // implicite : OK en C // En C++ ce serait : int *p = (int *)malloc(sizeof(int)); int x = 42; char *c = (char *)&x; // cast explicite nécessaire (int* → char*)

Attention Caster des pointeurs vers des types incompatibles peut violer les règles d'aliasing strict et provoquer un comportement indéfini.

Q5 Comment déclarer un paramètre dont le type est un pointeur de fonction ? +

Syntaxe : type_retour (*nom)(type_param1, type_param2, ...)

// Pointeur vers une fonction prenant 2 int et retournant int int (*comparateur)(int, int); // Comme paramètre d'une fonction void trier(int *tab, int n, int (*comparer)(int, int)) { // utilise comparer comme callback } // Avec typedef pour clarifier typedef int (*t_comparateur)(int, int); void trier(int *tab, int n, t_comparateur comparer);

Astuce lecture Pour lire la déclaration, commencer par le nom : "comparateur est un pointeur vers une fonction qui prend 2 int et retourne int".

Q6 (Bonus) Comment déclarer une fonction dont le retour est un pointeur de fonction ? +

Syntaxe sans typedef (difficile à lire) :

int (*get_operation(char op))(int, int); // get_operation prend un char et retourne un pointeur // vers une fonction prenant 2 int et retournant int

Avec typedef (beaucoup plus lisible) :

typedef int (*t_op)(int, int); t_op get_operation(char op) { if (op == '+') return addition; if (op == '*') return multiplication; return NULL; }

Exemple dans la stdlib : signal(int sig, void (*handler)(int)) retourne l'ancien handler.

Q7 Comment utiliser un pointeur de fonction pour appeler la fonction pointée ? (2 syntaxes) +
int (*f)(int, int) = addition; // f pointe vers addition // Syntaxe 1 : appel indirect explicite (déréférencement) int resultat = (*f)(3, 4); // Syntaxe 2 : appel direct (le compilateur comprend) int resultat2 = f(3, 4);

Les deux syntaxes sont équivalentes. La syntaxe 2 est plus courante et lisible.

Q8 Comment affecter l'adresse d'une fonction à un pointeur de fonction ? (2 syntaxes) +
int addition(int a, int b) { return a + b; } int (*f)(int, int); // Syntaxe 1 : avec & f = &addition; // Syntaxe 2 : sans & (nom de fonction = adresse implicitement) f = addition;

Les deux sont équivalentes. En C, le nom d'une fonction se convertit implicitement en pointeur vers cette fonction (comme le nom d'un tableau se convertit en pointeur vers son premier élément).

Q10 Comment attendre la fin d'un thread ? (nom et paramètres) +
#include <pthread.h> int pthread_join(pthread_t tid, void **retval);
  • tid : identifiant du thread à attendre
  • retval : si non NULL, reçoit la valeur retournée par le thread (celle passée à return ou pthread_exit())
  • Retour : 0 si succès, code d'erreur sinon

Q11 Différence principale entre processus fils et thread (mémoire partagée) ? +

Différence fondamentale : la mémoire

  • Processus fils (fork) : obtient une copie indépendante de l'espace mémoire du père (mécanisme Copy-On-Write). Les modifications d'une variable par le fils n'affectent pas le père.
  • Thread : partage la même mémoire que le thread créateur (variables globales, tas). Les modifications d'une variable partagée par un thread sont immédiatement visibles par les autres threads.

C'est pourquoi les threads nécessitent des mécanismes de synchronisation (mutex, sémaphores) pour éviter les race conditions, alors que les processus n'en ont pas besoin pour leur propre mémoire.

Q3 Qu'est-ce que l'indéterminisme pour une fonction ou un programme ? +

Un programme ou une fonction est indéterministe si son résultat peut varier d'une exécution à l'autre pour les mêmes entrées, en raison de facteurs non contrôlés.

Sources d'indéterminisme :

  • L'ordonnancement des threads/processus par le système (timing)
  • Les interruptions matérielles
  • La lecture de variables non initialisées
  • Les nombres aléatoires
  • Les accès concurrents non synchronisés (race conditions)

Un programme concurrent sans synchronisation est généralement indéterministe — et c'est un problème.

Q4 Qu'est-ce qu'une situation de compétition (race condition) ? +

Une race condition survient quand deux tâches accèdent à une ressource partagée de façon concurrente et que le résultat dépend de l'ordre d'exécution.

// Exemple : deux threads incrémentent un compteur int compteur = 0; void *incrementer(void *arg) { for (int i = 0; i < 100000; i++) compteur++; // ← race condition ! (read-modify-write non atomique) return NULL; }

Entre processus : race condition possible sur des fichiers partagés, segments de mémoire partagée, ou ressources système.

Entre threads : toute variable globale ou sur le tas accessible à plusieurs threads sans synchronisation est une race condition potentielle.

Q5 Qu'est-ce qu'un appel système bloquant ? Exemples ? +

Un appel système bloquant suspend l'exécution du processus (ou thread) jusqu'à ce que l'opération puisse être complétée. Le processus passe à l'état "sleeping" et libère le CPU.

Exemples :

  • read() sur un terminal ou un pipe vide : attend une saisie
  • wait() / waitpid() : attend la mort d'un fils
  • sem_wait() : attend que le sémaphore soit > 0
  • accept() sur un socket : attend une connexion entrante
  • pthread_join() : attend la fin d'un thread
  • nanosleep(), sleep() : attend un délai
Q6 Qu'est-ce qu'une instruction atomique ? Pourquoi s'en préoccuper ? +

Une instruction atomique est une opération indivisible : elle s'exécute entièrement ou pas du tout, sans possibilité d'interruption.

Pourquoi s'en préoccuper :

Des opérations semblant simples en C ne sont pas atomiques au niveau machine :

x++; // compile en 3 instructions : // 1. LOAD x → registre // 2. ADD registre + 1 // 3. STORE registre → x

Si deux threads font x++ simultanément sur x=0, on peut obtenir 1 au lieu de 2 (race condition).

Solutions : sémaphores, mutex, ou instructions atomiques matérielles (ex : __sync_fetch_and_add, <stdatomic.h> en C11).

Q7 Comment initialiser un sémaphore anonyme ? À quoi sert le 3ème paramètre ? +
#include <semaphore.h> int sem_init(sem_t *sem, int pshared, unsigned int value);
  • sem : pointeur vers le sémaphore à initialiser
  • pshared : 0 = sémaphore partagé entre threads du même processus | non-zéro = partagé entre processus (doit être en mémoire partagée)
  • value : valeur initiale du compteur (ex : 1 pour un sémaphore binaire)
  • Retour : 0 si succès, −1 si erreur
sem_t mutex_sem; sem_init(&mutex_sem, 0, 1); // sémaphore binaire pour threads
Q8 Comment détruire un sémaphore anonyme ? +
#include <semaphore.h> int sem_destroy(sem_t *sem);

Libère les ressources associées au sémaphore. À appeler quand le sémaphore n'est plus utilisé (après que tous les threads l'aient terminé).

Attention Détruire un sémaphore sur lequel des threads sont bloqués est un comportement indéfini.

Auto-évaluation 5 · Après CM5
Mutex, Deadlocks et Problèmes classiques
Exclusion mutuelle, étreintes mortelles, producteurs-consommateurs
Q1 Qu'est-ce qu'une section critique ? +

Une section critique est une portion de code qui accède à une ressource partagée (variable, fichier, structure de données…) et qui ne peut être exécutée que par un thread à la fois.

L'objectif est de garantir que les opérations sur la ressource partagée se déroulent sans interférence entre threads.

// Section critique : accès à "compteur" pthread_mutex_lock(&mutex); /* --- DÉBUT section critique --- */ compteur++; /* --- FIN section critique --- */ pthread_mutex_unlock(&mutex);

Propriétés souhaitées : exclusion mutuelle, absence de deadlock, absence de famine, efficacité (ne pas bloquer si la ressource est libre).

Q2 Qu'est-ce qu'un mutex ? Comment le créer, verrouiller, déverrouiller, détruire ? +

Un mutex (MUTual EXclusion) est un verrou qui garantit qu'un seul thread peut être dans la section critique à la fois.

#include <pthread.h> pthread_mutex_t m; // Créer (initialiser) pthread_mutex_init(&m, NULL); // Ou statiquement : pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER; // Verrouiller (lock) — bloque si déjà verrouillé pthread_mutex_lock(&m); // ... section critique ... // Déverrouiller (unlock) pthread_mutex_unlock(&m); // Détruire pthread_mutex_destroy(&m);

Variante non bloquante : pthread_mutex_trylock() retourne immédiatement avec une erreur si le mutex est déjà pris.

Q3 Contraintes supplémentaires d'un mutex vs un sémaphore binaire ? +

Un sémaphore peut être V() par un thread différent de celui qui a fait P(). Un mutex a des contraintes supplémentaires :

  • Propriété : le mutex doit être déverrouillé par le même thread qui l'a verrouillé. Déverrouiller un mutex depuis un autre thread est un comportement indéfini.
  • Non récursif par défaut : si un thread essaie de re-verrouiller un mutex qu'il détient déjà, c'est un deadlock (sauf avec PTHREAD_MUTEX_RECURSIVE).
  • Ces contraintes permettent des optimisations et une meilleure détection des erreurs.
Q4 Transmettre l'adresse d'un mutex à un thread (3 cas) ? +

a) Mutex global : accessible directement par tous les threads sans passer d'argument.

pthread_mutex_t mutex_global; // variable globale void *thread_fn(void *arg) { pthread_mutex_lock(&mutex_global); // accès direct ... }

b) Mutex dans la pile de main : passer son adresse via l'argument void *.

void *thread_fn(void *arg) { pthread_mutex_t *m = (pthread_mutex_t *)arg; pthread_mutex_lock(m); } int main() { pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER; pthread_create(&tid, NULL, thread_fn, &m); }

c) Mutex sur le tas dans une structure : structurer les données partagées avec le mutex inclus.

typedef struct { pthread_mutex_t mutex; int compteur; char *buffer; } t_donnees_partagees; void *thread_fn(void *arg) { t_donnees_partagees *d = (t_donnees_partagees *)arg; pthread_mutex_lock(&d->mutex); d->compteur++; pthread_mutex_unlock(&d->mutex); }
Q5 Qu'est-ce que le problème des philosophes ? +

Problème classique de synchronisation posé par Dijkstra :

  • 5 philosophes assis autour d'une table ronde, 5 fourchettes (une entre chaque paire)
  • Un philosophe pense, puis veut manger, puis pense à nouveau
  • Pour manger, un philosophe a besoin des deux fourchettes adjacentes
  • Si chacun prend la fourchette à sa gauche et attend celle à droite → deadlock !

Le problème illustre les défis de l'allocation de ressources partagées sans deadlock ni famine.

Solutions : numéroter les fourchettes et les prendre toujours dans l'ordre croissant, ou désigner un "arbitre", ou limiter à N−1 philosophes pouvant manger simultanément.

Q6 Qu'est-ce qu'une étreinte mortelle (deadlock) ? +

Un deadlock se produit quand deux (ou plus) threads se bloquent mutuellement en attendant des ressources que l'autre détient, sans qu'aucun ne puisse progresser.

// Thread A // Thread B lock(mutex1); lock(mutex2); lock(mutex2); ←→ lock(mutex1); // Deadlock ! A attend mutex2 (tenu par B), B attend mutex1 (tenu par A)

4 conditions de Coffman (toutes nécessaires pour un deadlock) :

  • Exclusion mutuelle : les ressources sont non partageables
  • Rétention et attente : un thread tient une ressource et en attend une autre
  • Non-préemption : les ressources ne peuvent être retirées de force
  • Attente circulaire : cycle dans le graphe d'allocation

Prévention : imposer un ordre total sur l'acquisition des ressources (toujours prendre mutex1 avant mutex2).

Q7 Problème producteurs-consommateurs : qu'est-ce que c'est ? Comment le résoudre ? +

Description : des producteurs ajoutent des éléments dans un buffer borné, des consommateurs les retirent. Contraintes : ne pas lire si buffer vide, ne pas écrire si buffer plein.

#define N 10 sem_t vide; // compte les places libres (init: N) sem_t plein; // compte les éléments disponibles (init: 0) pthread_mutex_t mutex; // accès exclusif au buffer // Producteur void *produire(void *arg) { while (1) { int item = produire_item(); sem_wait(&vide); // attendre une place libre pthread_mutex_lock(&mutex); inserer_dans_buffer(item); pthread_mutex_unlock(&mutex); sem_post(&plein); // signaler un élément ajouté } } // Consommateur void *consommer(void *arg) { while (1) { sem_wait(&plein); // attendre un élément pthread_mutex_lock(&mutex); int item = retirer_du_buffer(); pthread_mutex_unlock(&mutex); sem_post(&vide); // signaler une place libérée utiliser_item(item); } }
Q8 Problème lecteurs-écrivains : qu'est-ce que c'est ? Comment le résoudre ? +

Description : plusieurs lecteurs peuvent lire simultanément, mais un écrivain doit avoir l'accès exclusif (pas d'autre lecteur ni écrivain simultané).

int nb_lecteurs = 0; pthread_mutex_t mutex_lecteurs; // protège nb_lecteurs sem_t acces_ecriture; // exclusivité pour les écrivains // Lecteur void *lire(void *arg) { pthread_mutex_lock(&mutex_lecteurs); nb_lecteurs++; if (nb_lecteurs == 1) // premier lecteur : bloquer les écrivains sem_wait(&acces_ecriture); pthread_mutex_unlock(&mutex_lecteurs); // --- lecture de la ressource --- pthread_mutex_lock(&mutex_lecteurs); nb_lecteurs--; if (nb_lecteurs == 0) // dernier lecteur : libérer les écrivains sem_post(&acces_ecriture); pthread_mutex_unlock(&mutex_lecteurs); } // Écrivain void *ecrire(void *arg) { sem_wait(&acces_ecriture); // accès exclusif // --- écriture --- sem_post(&acces_ecriture); }

Attention Cette solution favorise les lecteurs : si des lecteurs arrivent continuellement, les écrivains peuvent souffrir de famine.

Q9 Qu'est-ce qu'une situation de famine (starvation) ? +

La famine (starvation) survient quand un thread (ou processus) est empêché d'avancer indéfiniment, non pas à cause d'un deadlock, mais parce que d'autres threads sont toujours prioritaires et monopolisent les ressources.

Exemple : dans le problème lecteurs-écrivains (solution favorisant les lecteurs), si des lecteurs arrivent en permanence, un écrivain peut attendre indéfiniment — c'est de la famine.

Différence avec deadlock :

  • Deadlock : personne ne progresse (blocage mutuel)
  • Famine : certains progressent, d'autres jamais

Solutions : files d'attente FIFO, politiques équitables (aging : augmenter la priorité avec le temps d'attente).

Q1 Quels sont les 4 appels système Unix fondamentaux pour les entrées-sorties ? +
  • open() : ouvre ou crée un fichier → retourne un descripteur de fichier
  • read() : lit des octets depuis un descripteur
  • write() : écrit des octets vers un descripteur
  • close() : ferme un descripteur de fichier

Ces 4 appels fonctionnent sur tout ce qui est représenté comme un fichier Unix : fichiers ordinaires, tubes (pipes), sockets, terminaux, périphériques…

Q2 Qu'est-ce qu'un descripteur de fichier (file descriptor) ? +

Un descripteur de fichier (fd) est un entier non négatif qui identifie un fichier ouvert dans le processus. C'est un index dans la table des fichiers ouverts du processus, maintenue par le noyau.

  • Valeurs réservées : 0 = stdin, 1 = stdout, 2 = stderr
  • Quand on open(), le noyau retourne le plus petit fd disponible ≥ 3
  • Le fd n'est valide que dans le processus qui l'a ouvert (sauf héritage par fork)

Abstraction puissante : le même code utilisant un fd peut lire depuis un fichier, un pipe ou une socket.

Q3 Que sont les types size_t et ssize_t ? +
  • size_t : entier non signé, suffisamment grand pour représenter la taille de n'importe quel objet en mémoire. Utilisé pour les tailles et les comptes. Défini dans <stddef.h>. Vaut typiquement 64 bits sur x86-64.
  • ssize_t : entier signé, même taille que size_t. Permet de retourner une taille ou une erreur (−1). Utilisé par read() et write().
ssize_t n = read(fd, buf, 1024); if (n == -1) { // erreur perror("read"); } else if (n == 0) { // fin de fichier printf("EOF\n"); } else { // n octets lus (1 ≤ n ≤ 1024) ... }
Q4 Que retourne open ? Comment ouvrir en lecture seule ? En écriture ? +

  • Échec : retourne −1 et met errno à jour

Q5 Fichier ouvert en écriture : créer, tronquer, ou écrire à la fin ? +

Q6 Quand open crée un fichier, à quoi sert le dernier argument ? +

Le 3ème argument est le mode : les permissions du fichier créé, exprimées en octal.

open("f.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644); // ^^^^ permissions

Ces permissions sont combinées avec le umask du processus : permissions_réelles = mode & ~umask.

Le 3ème argument est ignoré si O_CREAT n'est pas dans les flags.

Q7 & 8 Permissions Unix : signification et représentation en octal ? +

Chaque fichier a 9 bits de permissions : 3 groupes × 3 bits (rwx) pour propriétaire, groupe, autres.

Permissions : rwxrwxrwx │││││││││ │││││││└─ autres : exécuter (x) = 1 ││││││└── autres : écrire (w) = 2 │││││└─── autres : lire (r) = 4 ││││└──── groupe : exécuter │││└───── groupe : écrire ││└────── groupe : lire │└─────── proprio : exécuter └──────── proprio : écrire / lire 0755 = rwxr-xr-x (proprio: tout, groupe+autres: lecture+exécution) 0644 = rw-r--r-- (proprio: lire+écrire, groupe+autres: lecture seule) 0600 = rw------- (proprio seul peut lire/écrire) 0777 = rwxrwxrwx (tout le monde peut tout faire)

Mémo : r=4, w=2, x=1. Additionner pour chaque groupe.

Q9 Comment utiliser read ? (arguments, valeur de retour) +

  • buf : tampon de réception
  • count : nombre maximum d'octets à lire

Valeur de retour :

  • > 0 : nombre d'octets effectivement lus (peut être < count !)
  • 0 : fin de fichier (EOF)
  • -1 : erreur (errno mis à jour)

Q10 Dans quels cas read est-il bloquant ? +
  • Terminal (stdin) : attend que l'utilisateur tape une entrée et appuie sur Entrée
  • Pipe vide : attend qu'un processus écrive dans le pipe
  • Socket réseau : attend que des données arrivent
  • Fichier sur réseau (NFS) : peut bloquer si le serveur est lent

Pour les fichiers ordinaires sur disque, read est techniquement non-bloquant (il peut mettre en attente pendant l'accès disque, mais revient toujours).

Q11 Comment utiliser write ? (arguments, valeur de retour) +

  • buf : données à écrire
  • count : nombre d'octets à écrire

Valeur de retour :

  • > 0 : nombre d'octets effectivement écrits (peut être < count )
  • -1 : erreur

Q12 Les octets nuls dans les chaînes ont-ils une signification pour read/write ? +

Non. Les octets nuls ('\0') n'ont aucune signification particulière pour read et write. Ces appels système travaillent sur des séquences d'octets brutes, pas sur des chaînes C.

C'est une différence fondamentale avec les fonctions C de haut niveau comme printf ou strcpy qui s'arrêtent au premier '\0'.

Cela signifie que write peut écrire des données contenant des zéros (fichiers binaires, images, etc.) sans problème.

Ces fd sont ouverts automatiquement à la création du processus. Ils peuvent être redirigés par le shell (./prog > fichier).

Q14 Avec les entrées/sorties standard : lire une ligne, un caractère, données formatées ? +
// Lire une ligne d'entrée char buf[256]; fgets(buf, sizeof(buf), stdin); // depuis stdin fgets(buf, sizeof(buf), fichier); // depuis un fichier // Écrire une ligne dans un fichier fprintf(fichier, "Ligne\n"); fputs("Ligne\n", fichier); // Lire un caractère (octet) en entrée int c = fgetc(stdin); int c2 = getchar(); // équivalent fgetc(stdin) // Écrire un caractère en sortie fputc('A', stdout); putchar('A'); // équivalent fputc(stdout) // Lire données formatées int n; float f; fscanf(stdin, "%d %f", &n, &f); scanf("%d", &n); // équivalent fscanf(stdin) // Écrire données formatées fprintf(stdout, "%d %f\n", n, f); printf("%d\n", n); // équivalent fprintf(stdout)
Q15 Pourquoi getchar retourne un int et pas un char ? +

Parce que getchar() doit pouvoir signaler la fin de fichier (EOF).

EOF est défini comme −1 dans la bibliothèque standard. Or, un char non signé ne peut pas représenter −1 (il vaut 255 à la place). Et même un char signé aurait une ambiguïté avec le caractère d'octet 0xFF.

En retournant un int, on peut distinguer :

int c = getchar(); if (c == EOF) { // -1 : fin de fichier ou erreur printf("Fin\n"); } else { // c est un octet valide (0 à 255) printf("Caractère : %c\n", (char)c); }

Piège classique char c = getchar(); est incorrect : si char est non signé, la comparaison c == EOF est toujours fausse.

Q16 Quels sont les flux associés aux 3 canaux standards ? +
Flux C fd Description stdin 0 Entrée standard (type FILE*) stdout 1 Sortie standard (type FILE*) stderr 2 Sortie erreur (type FILE*)

Ces flux sont des variables globales de type FILE *, déclarées dans <stdio.h>. Ils encapsulent les descripteurs correspondants avec la gestion de la bufferisation.

Q17 Qu'est-ce que la bufferisation ? Les 3 modes ? Comportement par défaut ? +

La bufferisation accumule les données dans un tampon avant de les envoyer réellement au noyau via write(), pour réduire le nombre coûteux d'appels système.

3 modes de bufferisation :

  • Non bufferisé (_IONBF) : chaque écriture appelle write() immédiatement. Aucun tampon. Utilisé pour stderr.
  • Bufferisation par ligne (_IOLBF) : le tampon est vidé à chaque '\n' ou quand il est plein. Utilisé par stdout quand connecté à un terminal.
  • Bufferisation en blocs (_IOFBF) : le tampon n'est vidé que lorsqu'il est plein, ou lors d'un fflush(), ou à la fermeture. Utilisé pour les fichiers sur disque.

Comportements par défaut :

stderr → non bufferisé (messages d'erreur immédiats) stdout → par ligne (terminal) ou en blocs (fichier redirigé) fichiers → en blocs

Piège printf("msg") sans \n peut ne pas s'afficher si stdout est bufferisé par ligne. Utiliser fflush(stdout) pour forcer.

Thème 1 — Bases C et Représentation des données

Compilation

Le cycle de compilation en C : préprocesseur → compilateur → éditeur de liens. Les drapeaux essentiels sont -Wall -Wextra -g (avertissements et débogage) et -std=c99 pour le standard. Valgrind est l'outil de détection des fuites mémoire et accès invalides.

Représentation des données

La mémoire est byte addressable : chaque octet a une adresse unique. Un bit = 0 ou 1, un octet = 8 bits. Sur n bits, on représente 2ⁿ valeurs. Approximation clé : 2¹⁰ ≈ 10³.

Entiers signés (complément à 2, positifs et négatifs) vs non signés (positifs seulement). Conversions : divisions euclidiennes successives. Regroupement par 4 bits pour le passage binaire ↔ hexadécimal.

Types à taille fixe (C99)

<stdint.h> définit int8_t, uint32_t, etc. pour des tailles garanties indépendantes de l'architecture.

Boutisme (Endianness)

Little-endian (x86) : octet de poids faible en première adresse. Big-endian (réseau) : octet de poids fort en premier. Crucial pour la communication réseau et les fichiers binaires.

Thème 2 — Système d'exploitation et Processus

Noyau (Kernel)

Le noyau gère les processus, la mémoire, les E/S, le système de fichiers, les IPC et la sécurité. Il s'exécute en mode privilégié. Les programmes y accèdent via les appels système (syscalls), consultables avec man 2.

Processus

Un processus = un programme en cours d'exécution. Identifié par son PID. Attributs : PID, PPID, UID/GID, espace mémoire, fichiers ouverts, environnement, répertoire courant. La table de processus forme une arborescence dont la racine est init/systemd (PID 1).

Création de processus : fork()

fork() crée une copie du processus courant. Retourne 0 au fils, le PID du fils au père, −1 en cas d'erreur. Les deux processus poursuivent après le fork(), d'où la nécessité de tester la valeur de retour.

Arguments et environnement

Accès aux arguments via argc/argv. Variables d'environnement accessibles via getenv(), environ, ou le 3ème paramètre de main. Le PATH liste les répertoires de recherche des exécutables.

Signaux

Mécanisme de notification asynchrone. Peuvent terminer, ignorer, suspendre/reprendre un processus ou déclencher un handler. Envoi : kill(pid, sig). Terminaux génèrent SIGINT (Ctrl+C), SIGTSTP (Ctrl+Z).

Remplacement d'image : exec()

La famille exec() remplace le programme courant (même PID, nouveau code). execl/execv : chemin absolu requis. execlp/execvp : recherche dans PATH. Ne retourne jamais en cas de succès.

Attente : wait()/waitpid()

waitpid(pid, &status, opts) attend la mort d'un fils. Macros WIFEXITED/WEXITSTATUS pour analyser la terminaison. Sans wait(), un processus terminé devient zombie.

🔀
Thème 3 — Généricité : void* et Pointeurs de fonctions

void* — Pointeur générique

void * permet d'écrire des fonctions indépendantes du type concret. Il peut recevoir n'importe quel pointeur implicitement (en C). On ne peut pas le déréférencer ni faire d'arithmétique directement dessus — un cast vers le type concret est nécessaire. Utilisé par malloc, memcpy, qsort…

Pointeurs de fonctions

Syntaxe : type_retour (*nom)(params). Permettent de passer des callbacks à des fonctions génériques (comme qsort prend une fonction de comparaison). Deux syntaxes équivalentes pour affecter et appeler :

  • Affecter : f = ma_fonction ou f = &ma_fonction
  • Appeler : f(args) ou (*f)(args)

L'utilisation de typedef améliore grandement la lisibilité.

🧵
Thème 4 — Threads POSIX (pthreads)

Création et attente

pthread_create(&tid, NULL, fonction, arg) crée un thread qui exécute fonction(arg). La fonction est de type void *(*)(void *). pthread_join(tid, &retval) attend la fin du thread et récupère sa valeur de retour. Compiler avec -lpthread.

Threads vs Processus

Différence fondamentale : les threads partagent la mémoire (variables globales, tas) du processus, contrairement aux processus fils qui obtiennent une copie indépendante. Avantage : communication rapide. Inconvénient : nécessite synchronisation.

Passer des données à un thread

Via l'argument void * de pthread_create : passer l'adresse d'une variable, d'une structure, ou d'un tableau alloué dynamiquement. Ne jamais passer l'adresse d'une variable locale susceptible de sortir de portée.

🔒
Thème 5 — Synchronisation : Sémaphores et Mutex

Le problème : race conditions

Quand plusieurs threads accèdent à une ressource partagée sans synchronisation, le résultat dépend de l'ordonnancement → indéterminisme. Même une simple opération x++ est non atomique (3 instructions machine).

Sémaphores POSIX

Compteur entier avec deux opérations atomiques : sem_wait() (décrémente ou bloque si 0) et sem_post() (incrémente, réveille un thread bloqué). Initialisation : sem_init(&s, 0, valeur). Destruction : sem_destroy(&s).

Mutex POSIX

Verrou d'exclusion mutuelle. Contrainte supplémentaire vs sémaphore binaire : doit être déverrouillé par le même thread qui l'a verrouillé. Opérations : pthread_mutex_lock / unlock / destroy. Initialisation statique avec PTHREAD_MUTEX_INITIALIZER.

Deadlock et Famine

Deadlock : cycle d'attente mutuelle → aucun thread ne progresse. Prévention : imposer un ordre total sur l'acquisition des ressources. Famine : un thread attend indéfiniment parce que d'autres sont toujours prioritaires.

Problèmes classiques

  • Philosophes : deadlock si chacun prend une fourchette et attend l'autre
  • Producteurs-Consommateurs : buffer borné avec sémaphores vide/plein + mutex
  • Lecteurs-Écrivains : lecteurs simultanés possibles, écrivain exclusif ; risque de famine pour les écrivains
📁
Thème 6 — Entrées/Sorties Unix

Appels système de bas niveau

Les 4 syscalls fondamentaux : open(), read(), write(), close(). Fonctionnent sur tous les fichiers Unix : fichiers ordinaires, pipes, sockets, terminaux. Travaillent sur des octets bruts (pas de sens particulier au '\0').

Descripteurs de fichiers

Entier ≥ 0 identifiant un fichier ouvert. Réservés : 0 (stdin), 1 (stdout), 2 (stderr). open() retourne le plus petit fd libre ≥ 3. ssize_t permet de distinguer les données (≥ 0) des erreurs (−1).

Flags d'ouverture

O_RDONLY, O_WRONLY, O_RDWR pour le mode. O_CREAT | O_TRUNC pour créer/vider, O_APPEND pour écrire à la fin. Le mode de permission (ex : 0644) est requis avec O_CREAT.

Permissions Unix

9 bits : rwx pour propriétaire, groupe, autres. Mémo : r=4, w=2, x=1. 0644 = rw-r--r-- (fichier standard), 0755 = rwxr-xr-x (exécutable).

Bibliothèque stdio (E/S standard)

Couche de haut niveau au-dessus des syscalls : fopen/fclose, fgets/fputs, fgetc/fputc, fprintf/fscanf. getchar() retourne int (pas char) pour pouvoir représenter EOF (−1).

Bufferisation

3 modes : non bufferisé (stderr), par ligne (stdout terminal), en blocs (fichiers). La bufferisation accumule les données pour réduire les appels système. Forcer le vidage avec fflush(). Piège : printf sans \n peut ne pas s'afficher immédiatement en mode ligne.
