Exercice 1 Design pattern Itérateur 5 points
Question 1 — SalleIterator 2 pts

On crée une classe qui mémorise le tableau, sa taille, et un curseur index qui avance à chaque appel de next().

import java.util.Iterator;

public class SalleIterator implements Iterator<Relique> {
    private Relique[] tab;
    private int size;
    private int index;

    public SalleIterator(Relique[] tab, int size) {
        this.tab   = tab;
        this.size  = size;
        this.index = 0;          // le curseur commence au début
    }

    @Override
    public boolean hasNext() {
        return index < size;     // il reste des éléments si le curseur n'a pas atteint la fin
    }

    @Override
    public Relique next() {
        return tab[index++];     // renvoie l'élément courant et avance le curseur
    }
}
Pourquoi index++ et pas ++index ? tab[index++] renvoie d'abord tab[index], puis incrémente. Avec tab[++index] on incrémenterait avant et on sauterait le premier élément.
Question 2 — Méthode iterator() 1 pt

On instancie simplement un SalleIterator en lui passant le tableau interne et le nombre d'éléments.

// Dans la classe Salle :
@Override
public Iterator<Relique> iterator() {
    return new SalleIterator(reliques, nbReliques);
}
Pourquoi pas new SalleIterator(reliques, reliques.length) ? Le tableau a une capacité de 20, mais seules nbReliques cases sont remplies. Passer reliques.length ferait itérer sur les cases vides (null).
Question 3 — For amélioré avec filtre 2 pts
Salle s = ...;

for (Relique r : s) {                     // for amélioré possible car Salle implements Iterable
    if (r.getPuissance() > 30) {
        System.out.println(r.getNom());
    }
}
Mécanisme Le compilateur Java traduit for (Relique r : s) en appel de s.iterator(), puis boucle sur hasNext() / next(). C'est exactement pour ça qu'on a fait implémenter Iterable<Relique> à Salle.
Exercice 2 Design patterns Singleton & Enum 4 points
Question 1 — Enum TypeGardien 2 pts
public enum TypeGardien {
    SQUELETTE(10),
    VAMPIRE(50),
    SPECTRE(90);

    private final int dangerosité;

    // Le constructeur d'un enum est TOUJOURS private (implicitement ou explicitement)
    private TypeGardien(int dangerosité) {
        this.dangerosité = dangerosité;
    }

    public int getDangerosité() {
        return dangerosité;
    }
}
Points clés
  • Les valeurs de l'enum (SQUELETTE, etc.) sont déclarées en premier, séparées par des virgules, terminées par un point-virgule avant les attributs.
  • Le constructeur est toujours privé pour un enum — c'est obligatoire.
  • Les attributs sont final car un enum doit être non modifiable (immuable).
  • Les valeurs de dangerosité sont libres, l'essentiel est qu'elles soient cohérentes (croissantes ici).
Erreur fréquente Oublier le point-virgule après la dernière valeur (SPECTRE(90);) avant de déclarer les attributs. C'est une erreur de compilation.
Question 2 — Singleton JournalChateau 2 pts
public class JournalChateau {

    // 1. L'unique instance, créée une seule fois, non modifiable
    public static final JournalChateau theInstance = new JournalChateau();

    // 2. Constructeur PRIVÉ : personne d'autre ne peut créer d'instance
    private JournalChateau() {
        // rien à initialiser ici
    }

    // 3. Interdire explicitement le clonage (même pour les sous-classes)
    @Override
    protected final JournalChateau clone() throws CloneNotSupportedException {
        throw new CloneNotSupportedException();
    }

    // 4. La méthode métier
    public void enregistrer(String message) {
        System.out.println("[LOG] " + message);
    }
}

Utilisation :

JournalChateau.theInstance.enregistrer("Le chasseur entre dans la salle Nord.");
// Affiche : [LOG] Le chasseur entre dans la salle Nord.
Pourquoi interdire clone() ? Sans cette précaution, n'importe quelle sous-classe pourrait appeler super.clone() et créer une seconde instance, contournant le Singleton. Le final sur la méthode empêche aussi une sous-classe de la redéfinir.
Exercice 3 Design pattern Strategy & Lambda expressions 5 points
Questions 1 & 2 — StrategieMaxReliques + Singleton 2 pts
public class StrategieMaxReliques implements StrategieExploration {

    // Singleton : une seule instance, accessible via INSTANCE
    public static final StrategieMaxReliques INSTANCE = new StrategieMaxReliques();

    // Constructeur privé
    private StrategieMaxReliques() {}

    @Override
    public Salle choisirSalle(Salle p, Predicate<? super Relique> pred) {
        Salle meilleure = null;
        int max = 0;

        for (Direction d : Direction.values()) {
            Salle voisine = p.salleSuivante(d);

            if (voisine != null) {
                // Compter les reliques satisfaisant le prédicat dans cette salle
                int count = 0;
                for (Relique r : voisine) {     // possible car Salle implements Iterable
                    if (pred.test(r)) {
                        count++;
                    }
                }
                // On garde la salle si elle a strictement plus de reliques valides
                if (count > max) {
                    max = count;
                    meilleure = voisine;
                }
            }
        }

        return meilleure; // null si aucune salle voisine, ou si toutes ont count == 0
    }
}
Justification du Singleton StrategieMaxReliques n'a aucun état (pas d'attribut modifiable) : deux instances seraient strictement identiques. Une seule suffit, ce qui évite des allocations mémoire inutiles. C'est exactement le cas d'usage du Singleton décrit dans le cours (cf. Comparator).

Utilisation :

Salle prochaine = StrategieMaxReliques.INSTANCE.choisirSalle(
    salleCourante,
    r -> !r.estMaudite()
);
Question 3 — Lambda expressions 2 pts

a) Relique non maudite et puissance > 50 :

Predicate<Relique> pred = r -> !r.estMaudite() && r.getPuissance() > 50;

b) Relique non nulle dont le nom commence par "Or" :

Predicate<Relique> pred = r -> r != null && r.getNom().startsWith("Or");
Ordre des conditions On teste r != null en premier. Si on écrivait r.getNom().startsWith("Or") && r != null et que r est null, Java évaluerait quand même r.getNom() en premier → NullPointerException. Grâce à l'évaluation paresseuse de &&, si le premier membre est false, le second n'est jamais évalué.

c) Non maudite, puissance > 20, et nom commençant par "A" :

Predicate<Relique> pred = r -> !r.estMaudite()
                             && r.getPuissance() > 20
                             && r.getNom().startsWith("A");
Lisibilité On peut écrire une lambda sur plusieurs lignes pour la rendre lisible. La règle est : si l'expression est complexe, sauter des lignes est conseillé. Si elle tient bien en une ligne, une seule ligne est préférable.
Question 4 — Functional interface 1 pt

Predicate<E> est une functional interface car elle ne possède qu'une seule méthode abstraite (boolean test(E e)). Cela permet au compilateur Java d'accepter une lambda expression partout où un Predicate est attendu, sans avoir à créer une classe explicite : la lambda est automatiquement convertie en instance anonyme de l'interface.

Exercice 4 Design pattern Decorator 5 points
Question 1 — Implémentation complète de FilterIterator 2 pts
import java.util.Iterator;

public class FilterIterator<E> implements Iterator<E> {
    private Iterator<? extends E> supportIter;
    private Predicate<? super E> predicate;
    private boolean nextFound;   // true si nextElt contient un élément valide
    private E nextElt;           // prochain élément valide mis en réserve

    public FilterIterator(Iterator<? extends E> iter, Predicate<? super E> p) {
        this.supportIter = iter;
        this.predicate   = p;
        findNext();              // on prépare immédiatement le premier élément
    }

    // Cherche le prochain élément satisfaisant le prédicat dans l'itérateur source
    private void findNext() {
        nextFound = false;
        while (supportIter.hasNext()) {
            E candidate = supportIter.next();
            if (predicate.test(candidate)) {
                nextElt   = candidate;
                nextFound = true;
                return;          // trouvé, on s'arrête
            }
        }
        // Si on sort de la boucle sans avoir trouvé : nextFound reste false
    }

    @Override
    public boolean hasNext() {
        return nextFound;
    }

    @Override
    public E next() {
        E result = nextElt;  // on retourne l'élément mis en réserve
        findNext();          // et on prépare déjà le suivant
        return result;
    }
}
Pourquoi findNext() dans le constructeur ? Si on ne cherchait pas le premier élément à la construction, hasNext() renverrait false au premier appel (car nextFound vaut false par défaut), même si des éléments valides existent. Il faut "amorcer" l'itérateur.
Pourquoi stocker l'élément à l'avance (nextElt) ? Parce que pour répondre à hasNext() il faut déjà savoir si le prochain élément satisfait le prédicat. On ne peut pas le savoir sans consommer l'itérateur source. Stocker l'élément en avance permet de le "rendre" proprement au prochain appel de next().

Question 3 — Iterator<String> des noms, filtrés et transformés 1 pt
Salle s = ...;

// Étape 1 : itérateur de base sur les reliques de la salle
Iterator<Relique> iterBase = s.iterator();

// Étape 2 : on filtre (non maudites ET puissance > 40)
Iterator<Relique> iterFiltre = new FilterIterator<>(
    iterBase,
    r -> !r.estMaudite() && r.getPuissance() > 40
);

// Étape 3 : on transforme chaque Relique en son nom (String)
Iterator<String> iterNoms = new MappingIterator<>(
    iterFiltre,
    r -> r.getNom()      // Function<Relique, String>
);

// Utilisation
while (iterNoms.hasNext()) {
    System.out.println(iterNoms.next());
}
Ordre des opérations On filtre avant de transformer. Si on faisait l'inverse (transformer d'abord en String, puis filtrer), on n'aurait plus accès à estMaudite() ni à getPuissance() sur les chaînes.
Question 4 — Avantage de FilterIterator vs FilterSalleIterator 0.5 pt

FilterIterator fonctionne avec n'importe quel itérateur (sur un tableau, une liste, un set, ou même un autre FilterIterator), alors qu'un FilterSalleIterator ne pourrait filtrer que les reliques d'une Salle, obligeant à redéfinir une nouvelle classe filtrante pour chaque structure de données.

Question 5 — Deux FilterIterator empilés 0.5 pt
Salle s = ...;

// Premier filtre : reliques non maudites
Iterator<Relique> iter1 = new FilterIterator<>(
    s.iterator(),
    r -> !r.estMaudite()
);

// Deuxième filtre (appliqué SUR le premier) : nom commençant par "S"
Iterator<Relique> iter2 = new FilterIterator<>(
    iter1,
    r -> r.getNom().startsWith("S")
);

while (iter2.hasNext()) {
    System.out.println(iter2.next().getNom());
}
C'est ça le pattern Decorator ! On enveloppe un décorateur dans un autre décorateur. iter2 ne sait pas que iter1 est lui-même un FilterIterator. Il le traite comme n'importe quel Iterator<Relique>. On peut ainsi empiler autant de filtres/transformations que nécessaire.
Exercice 5 Architecture MVC & Design pattern Command 4 points
Question 1 — Classe PrendreRelique 1.5 pts
public class PrendreRelique implements Command {
    private Chasseur chasseur;
    private ListView<Relique> view; // on pourrait aussi la recréer dans execute()

    public PrendreRelique(Chasseur chasseur) {
        this.chasseur = chasseur;
    }

    @Override
    public void execute() {
        // 1. Construire et afficher la liste des reliques de la salle courante
        Salle salleCourante = chasseur.getSalleActuelle();
        ListView<Relique> menu = ListView.buildMenu(salleCourante);
        menu.show();

        // 2. Lire le choix de l'utilisateur
        Relique choisie = menu.readChoice();

        // 3. Le chasseur prend la relique choisie
        chasseur.prendreRelique(choisie);
    }
}
Respect du MVC PrendreRelique est une classe du Controller : elle fait appel à la View (ListView) pour l'affichage et le choix, et au Model (Chasseur) pour exécuter l'action. Elle ne fait aucun System.out.println directement.
Question 2 — Classe SeDeplacer 1.5 pts
import java.util.ArrayList;
import java.util.List;

public class SeDeplacer implements Command {
    private Chasseur chasseur;
    private ChateauMap carte;

    public SeDeplacer(Chasseur chasseur, ChateauMap carte) {
        this.chasseur = chasseur;
        this.carte    = carte;
    }

    @Override
    public void execute() {
        Salle salleCourante = chasseur.getSalleActuelle();

        // 1. Construire la liste des directions valides
        List<Direction> directionsValides = new ArrayList<>();
        for (Direction d : Direction.values()) {
            if (carte.salleExiste(salleCourante, d)) {
                directionsValides.add(d);
            }
        }

        // 2. Afficher les directions disponibles
        ListView<Direction> menu = ListView.buildMenu(directionsValides);
        menu.show();

        // 3. Lire le choix de direction
        Direction directionChoisie = menu.readChoice();

        // 4. Effectuer le déplacement
        carte.deplacer(chasseur, directionChoisie);
    }
}
Pourquoi une ArrayList et pas un tableau ? On ne sait pas à l'avance combien de directions sont valides (entre 0 et 4). Une ArrayList grandit dynamiquement. On aurait pu aussi utiliser un tableau de taille 4 et compter les valides, mais c'est plus lourd.
Erreur fréquente Passer Direction.values() directement à buildMenu afficherait toutes les directions, y compris celles sans salle voisine. Il faut bien filtrer d'abord avec salleExiste.
Question 3 — Apport du MVC 1 pt

La séparation MVC garantit que la View (l'affichage et la saisie) est totalement indépendante du Model (les données) et du Controller. On peut donc remplacer l'interface console (ListView en texte) par une interface graphique Swing (GUIView) sans modifier une seule ligne des classes du Model (Chasseur, Salle, etc.) ni des classes Command — il suffit de changer les implémentations de ListView.

Dans l'autre sens On peut aussi modifier les règles du jeu (comment fonctionne prendreRelique) sans toucher à l'affichage. Chaque couche peut évoluer indépendamment. C'est le principe fondamental du MVC.
