Introduction aux Design Patterns

C'est quoi un design pattern ?

Un design pattern (ou motif de conception) est une solution réutilisable à un problème de conception courant. Ce n'est pas du code copier-coller, c'est une recette : une façon bien connue de structurer ton code pour qu'il soit plus facile à maintenir et à faire évoluer.

Analogie Imagine que tu cuisines : un design pattern c'est comme une recette. Tu n'inventes pas à chaque fois comment faire une béchamel — tu suis la recette éprouvée. En programmation, pareil.

01

Le pattern Iterator

Comportement

Le problème à résoudre

Imagine que tu as une collection d'objets (une liste, un tableau, un ensemble...). Tu veux les parcourir tous, un par un. Le problème : selon la structure (ArrayList, LinkedList, tableau simple, Set...), la façon d'y accéder est différente.

L'itérateur résout ça en fournissant une interface unifiée pour parcourir n'importe quelle collection, sans avoir à savoir comment elle est codée en interne.

Analogie C'est comme une télécommande universelle. Peu importe si ta TV est Sony ou Samsung, la télécommande a toujours les boutons ▶ (suivant) et ◆ (est-ce qu'il reste quelque chose ?).

L'interface Iterator<E>

En Java, un itérateur implémente cette interface :

Iterator.java
public interface Iterator<E> {
    boolean hasNext();   // Reste-t-il un élément à lire ?
    E       next();       // Renvoie l'élément suivant ET avance le curseur
    void    remove();     // Supprime le dernier élément renvoyé (optionnel)
}
Attention ! Un itérateur ne peut servir qu'UNE SEULE FOIS. Une fois qu'on a parcouru tous les éléments, c'est terminé. Pour recommencer, il faut en créer un nouveau.

Le lien avec le for-each

Le for (String s : uneCollection) que tu utilises tout le temps, c'est exactement du sucre syntaxique pour :

Équivalence for-each ↔ Iterator
// Version for-each (ce qu'on écrit)
for (String s : uneListe) {
    System.out.println(s);
}

// Version Iterator (ce que Java fait vraiment)
Iterator<String> iter = uneListe.iterator();
while (iter.hasNext()) {
    String s = iter.next();
    System.out.println(s);
}

L'interface Iterable<E>

Pour qu'une classe soit utilisable dans un for-each, elle doit implémenter Iterable<E> :

Iterable.java
public interface Iterable<E> {
    Iterator<E> iterator();  // Renvoie un nouvel itérateur
}

Exemple concret : rendre une Pièce itérable (TD1)

Si la classe Piece stocke ses objets dans un tableau (pas dans une ArrayList), voici comment lui ajouter un itérateur :

Piece.java
public class Piece implements Iterable<ObjetZork> {
    private ObjetZork[] mesObjets;
    private int nbObjets;

    public Iterator<ObjetZork> iterator() {
        return new TableauIterator(
            mesObjets, nbObjets
        );
    }
}
TableauIterator.java
public class TableauIterator
    implements Iterator<ObjetZork> {

    private ObjetZork[] tab;
    private int nbElts;
    private int index = 0;

    public TableauIterator(
            ObjetZork[] t, int n) {
        tab = t; nbElts = n;
    }

    public boolean hasNext() {
        return index < nbElts;
    }

    public ObjetZork next() {
        return tab[index++];
    }
}
À l'examen — ce qu'on peut te demander Implémenter un itérateur custom (FilterNotNullIterator, FilterTransportableIterator...). La logique : hasNext() cherche le prochain élément valide, next() le renvoie et avance. La méthode findNext() (chercher le prochain index valide) est souvent le cœur de l'exercice.
02

Singleton & Enum

Création

Le problème : garantir une seule instance

Parfois, on veut qu'il n'existe qu'un seul objet d'une classe. Exemple : une classe Comparator sans état — pas besoin d'en créer 1000 exemplaires identiques. Le Singleton garantit qu'une seule instance sera créée.

Analogie Le Président de la République : il ne peut y en avoir qu'un à la fois. Si quelqu'un essaie d'en "créer" un nouveau, on lui dit non.

Les 3 règles du Singleton en Java

  1. Constructeur privé → personne d'autre ne peut créer d'instance
  2. Variable statique finale → stocke l'unique instance
  3. Accès via variable publique ou méthode statique → pour récupérer l'instance
Singleton — Version 1 : variable publique (recommandée en cours)
public class MonSingleton {
    // L'unique instance, créée une seule fois au chargement de la classe
    public static final MonSingleton theInstance
        = new MonSingleton();

    // Constructeur PRIVÉ : personne d'autre ne peut créer d'instance
    private MonSingleton() { }

    // Méthodes publiques normales...
    public void faireTruc() { ... }
}
Utilisation
// On accède à l'unique instance via theInstance
MonSingleton.theInstance.faireTruc();

// INTERDIT — ne compile pas car constructeur private
MonSingleton m = new MonSingleton(); // ❌ ERREUR !

Version 2 : méthode getInstance() (lazy initialization)

L'instance n'est créée que quand on en a besoin pour la première fois :

Singleton — Version 2 : getInstance()
public class MonSingleton {
    private static MonSingleton theInstance;  // null au départ

    private MonSingleton() { }

    public static MonSingleton getInstance() {
        if (theInstance == null) {         // Première fois ?
            theInstance = new MonSingleton();  // On crée l'instance
        }
        return theInstance;  // On renvoie toujours la même
    }
}

Les Enum en Java

Une enum est une extension naturelle du Singleton : c'est un ensemble fini de Singletons. En Java, chaque valeur d'une enum est garantie être une instance unique.

Enum simple
public enum Direction {
    NORTH, SOUTH, EAST, WEST
}
// Utilisation :
Direction d = Direction.NORTH;
Enum avec attributs et méthodes (comme ObjetZork)
public enum ObjetZork {
    // Les instances (Singletons) de l'enum
    EPEE("Épée", 5, true),
    TRESOR("Trésor", 20, false),
    POTION("Potion", 1, true);

    // Attributs (TOUJOURS finaux dans une enum)
    private final String nom;
    private final int poids;
    private final boolean transportable;

    // Constructeur PRIVÉ (implicitement)
    ObjetZork(String nom, int poids, boolean transportable) {
        this.nom = nom;
        this.poids = poids;
        this.transportable = transportable;
    }

    public String getNom()           { return nom; }
    public int getPoids()             { return poids; }
    public boolean estTransportable() { return transportable; }
}
Règle d'or des enum Une enum doit être NON MODIFIABLE (immutable). Ses attributs sont toujours final. Si tu mets des setters dans une enum, tu casses la garantie de Singleton.
À l'examen On peut te demander de transformer une classe ordinaire en enum (TD1, Ex.5), ou d'appliquer le Singleton à une classe Strategy/Comparator. La formule à retenir : constructeur private + static final theInstance.

Le pattern Strategy

Comportement

Le problème : choisir un algorithme à l'exécution

Tu as une classe qui doit faire quelque chose (trier, se déplacer, comparer...) de plusieurs façons différentes. Au lieu de coder tous les cas dans une seule méthode avec des if/else, on encapsule chaque algorithme dans sa propre classe.

Analogie GPS : plusieurs itinéraires possibles (le plus rapide, le plus court, éviter les péages). L'algorithme de calcul d'itinéraire est interchangeable. On choisit la "stratégie" au moment où on veut, sans changer le GPS lui-même.

Exemple canonique : Comparator<E>

Comparator est exactement le pattern Strategy. On peut fournir différentes stratégies de comparaison à list.sort() :

Strategy — Trier des ObjetZork
// Stratégie 1 : trier par poids
public class CompareWeight implements Comparator<ObjetZork> {
    public int compare(ObjetZork o1, ObjetZork o2) {
        return o1.getPoids() - o2.getPoids();
    }
}

// Stratégie 2 : trier par nom
public class CompareName implements Comparator<ObjetZork> {
    public int compare(ObjetZork o1, ObjetZork o2) {
        return o1.getNom().compareTo(o2.getNom());
    }
}

// Utilisation : on change la stratégie en changeant l'objet passé
List<ObjetZork> liste = ...;
liste.sort(new CompareWeight());  // tri par poids
liste.sort(new CompareName());    // tri par nom

Predicate<E> : une autre functional interface

Le pattern Strategy s'applique aussi avec Predicate<E>, qui est une interface avec une seule méthode : boolean test(E e).

Predicate — Tester des conditions
// Predicate : tester si un objet est non null
public class TestNotNull<E> implements Predicate<E> {
    public boolean test(E e) {
        return e != null;
    }
}

// Predicate : tester si un ObjetZork est transportable
public class TestTransportable implements Predicate<ObjetZork> {
    public boolean test(ObjetZork oz) {
        return oz.estTransportable();
    }
}

Les lambdas : Strategy sans créer de classe

Depuis Java 8, pour les interfaces fonctionnelles (une seule méthode abstraite), on peut utiliser une expression lambda au lieu de définir une classe entière. C'est beaucoup plus concis !

Lambdas — Syntaxe (paramètres) -> corps
// Sans lambda (verbose)
liste.sort(new CompareWeight());

// Avec lambda (concis)
liste.sort((o1, o2) -> o1.getPoids() - o2.getPoids());

// Predicate avec lambda
Predicate<ObjetZork> pTransp = oz -> oz.estTransportable();
Predicate<String> pNotNull = s -> s != null;
Predicate<ObjetZork> pLourd = oz -> oz.getPoids() > 10;

Application : la classe FilterArrayIterator (TD2)

Au lieu d'avoir une classe par type de filtre (FilterNotNullIterator, FilterTransportableIterator...), on crée une seule classe générique qui accepte un Predicate en paramètre :

FilterArrayIterator.java — Le pattern Strategy appliqué aux itérateurs
public class FilterArrayIterator<E> implements Iterator<E> {
    private E[] tab;
    private int nbElts;
    private Predicate<? super E> predicate;  // LA STRATÉGIE
    private int index;

    public FilterArrayIterator(E[] tab, int n, Predicate<? super E> p) {
        this.tab = tab;
        this.nbElts = n;
        this.predicate = p;
        index = findNext(0);  // chercher le 1er élément valide
    }

    private int findNext(int from) {
        while (from < nbElts && !predicate.test(tab[from])) {
            from++;
        }
        return from;
    }

    public boolean hasNext() { return index < nbElts; }

    public E next() {
        E elt = tab[index];
        index = findNext(index + 1);
        return elt;
    }
}
À l'examen Combiner Strategy + Singleton est très fréquent. Si une stratégie n'a pas d'état (pas d'attributs qui changent), on peut en faire un Singleton. Ex : StrategieMaxPredicate.INSTANCE.

Le pattern Decorator

Structure

Le problème : ajouter des fonctionnalités dynamiquement

On veut ajouter des comportements à un objet pendant l'exécution, sans toucher à sa classe d'origine. L'alternative serait l'héritage, mais ça explose le nombre de sous-classes (une par combinaison possible).

Analogie — La pizzeria Une pizza de base coûte 10€. On veut pouvoir ajouter des ingrédients (olives +2€, oignons +1€, mozzarella +3€...). Avec l'héritage, il faudrait une classe par combinaison : PizzaOlives, PizzaOignonsMozzarella, etc. → explosion ! Avec le Decorator, chaque ingrédient est un "enrobage" qui s'ajoute autour de la pizza.

Principe

  • Le Decorator implémente la même interface que l'objet qu'il décore
  • Il contient une référence vers l'objet décoré
  • Il délègue les appels à l'objet décoré, en ajoutant son propre comportement
  • On peut empiler plusieurs décorateurs
Decorator — La pizzeria (exemple du cours)
// Interface commune à tous (pizza + décorateurs)
public interface Pizza {
    String getDesc();
    int    getPrice();
}

// La pizza de base (l'objet décoré)
public class SimplePizza implements Pizza {
    public String getDesc()  { return "Pizza de base"; }
    public int    getPrice() { return 10; }
}

// Un décorateur : olives
public class PizzaAvecOlives implements Pizza {
    private Pizza pizza;  // Référence vers l'objet décoré

    public PizzaAvecOlives(Pizza p) { pizza = p; }

    public String getDesc()  { return pizza.getDesc() + " avec olives"; }
    public int    getPrice() { return pizza.getPrice() + 2; }
}

// Utilisation : on peut empiler les décorateurs !
Pizza p = new SimplePizza();
p = new PizzaAvecOlives(p);     // 12€ : "Pizza de base avec olives"
p = new PizzaAvecOignons(p);    // 13€ : "Pizza de base avec olives avec oignons"

Application au TD : FilterIterator (TD3)

Au TD2, FilterArrayIterator ne fonctionnait que sur des tableaux. Avec le Decorator, on crée FilterIterator qui fonctionne sur n'importe quel itérateur :

FilterIterator.java — Décorateur d'itérateur
public class FilterIterator<E> implements Iterator<E> {
    private Iterator<? extends E> supportIter; // L'itérateur décoré
    private Predicate<? super E> predicate;
    private boolean nextFound;
    private E nextElt;

    public FilterIterator(Iterator<? extends E> iter, Predicate<? super E> p) {
        supportIter = iter;
        predicate = p;
        findNext();  // Préparer le premier élément valide
    }

    private void findNext() {
        nextFound = false;
        while (supportIter.hasNext()) {
            E candidate = supportIter.next();
            if (predicate.test(candidate)) {
                nextElt = candidate;
                nextFound = true;
                return;
            }
        }
    }

    public boolean hasNext() { return nextFound; }

    public E next() {
        E result = nextElt;
        findNext();   // Préparer le suivant
        return result;
    }
}

// Utilisation — fonctionne sur TOUT itérateur (List, Set, tableau...)
List<String> list = Arrays.asList(null, "aa", null, "bb");
Iterator<String> iter = new FilterIterator<>(list.iterator(), s -> s != null);
// Affichera :aa:bb
Différence clé : FilterArrayIterator vs FilterIterator
• FilterArrayIterator (TD2, Strategy) : travaille sur un tableau, avance avec un index
• FilterIterator (TD3, Decorator) : enrobe n'importe quel Iterator, délègue hasNext/next à l'itérateur interne

Les Streams Java

Java 8+

C'est quoi un Stream ?

Un Stream<E> est une API Java 8 qui permet de faire des opérations en chaîne sur des collections de données (filtrer, transformer, compter...) de façon très lisible.

Opérations intermédiaires vs terminales

TypeCe que ça faitExemples
IntermédiaireProduit un nouveau Stream (calcul différé)filter(), map()
TerminaleDéclenche le calcul, produit un résultat finaliterator(), toList(), count(), anyMatch()

Après une opération terminale, le Stream est "consumé" et ne peut plus être utilisé.

Attention — Stream à usage unique ! Un Stream ne peut être parcouru qu'une fois. Si tu essaies de l'utiliser après une opération terminale → IllegalStateException.
Stream — Opérations de base
// Créer un Stream
Stream<Direction> s1 = Stream.of(Direction.values());

// filter() : garder seulement les éléments satisfaisant un prédicat
Stream<Piece> s2 = s1.map(d -> p.pieceSuivante(d))   // transformer
                      .filter(v -> v != null);              // filtrer

// Opérations terminales
List<Piece> pieces = s2.toList();          // → liste
long nb = s2.count();                       // → compter
boolean ok = s2.anyMatch(v -> v.hasOr()); // → au moins un satisfait ?
boolean all = s2.allMatch(v -> v != null);// → tous satisfont ?

Exemple — StrategieMaxPredicate avec Stream (TD4)

Trouver la pièce voisine contenant le plus d'objets satisfaisant un prédicat :

StrategieMaxPredicate — Version sans Stream puis avec
// Version sans Stream (avec boucles for — vue en cours)
public Piece choisirPiece(Piece p, Predicate<? super ObjetZork> pred) {
    Piece result = null;
    int max = 0;
    for (Direction d : Direction.values()) {
        Piece voisine = p.pieceSuivante(d);
        if (voisine != null) {
            int count = 0;
            for (ObjetZork oz : voisine) {
                if (pred.test(oz)) count++;
            }
            if (count > max) { max = count; result = voisine; }
        }
    }
    return result;
}

// Version avec Stream (plus concise)
public Piece choisirPiece(Piece p, Predicate<? super ObjetZork> pred) {
    return Stream.of(Direction.values())
        .map(d -> p.pieceSuivante(d))
        .filter(v -> v != null)
        .max((a, b) -> /* comparer count */)
        .orElse(null);
}

Command & Architecture MVC

Comportement + Architecture

L'architecture MVC

MVC = Model – View – Controller. C'est un principe qui sépare le code en 3 couches indépendantes :

CoucheRôleContrainte
Model Les données du jeu (Piece, Gamer, ObjetZork, ZorkMap...) Aucun affichage, aucune interaction utilisateur
View Les classes qui affichent/lisent (console, GUI, menus) Ne dépend pas du Model ni du Controller
Controller Commandes de l'utilisateur (GoCommand, TakeObject...) Appelle le Model pour exécuter, appelle la View pour afficher
Analogie Application bancaire : le Model = la base de données des comptes. La View = l'interface web que tu vois. Le Controller = ce qui traite ton clic "Virer 100€" en appelant la base ET en mettant à jour l'affichage.

Le design pattern Command

Chaque action de l'utilisateur est encapsulée dans une classe implémentant l'interface Command :

Command.java — L'interface
public interface Command {
    void execute();  // Exécuter la commande
}
RoomInventory.java — Afficher les objets de la pièce courante
public class RoomInventory implements Command {
    private Gamer gamer;
    private Game game;

    public RoomInventory(Gamer gamer, Game game) {
        this.gamer = gamer;
        this.game = game;
    }

    @Override
    public void execute() {
        // Récupérer la pièce courante (Model)
        Room room = gamer.getCurrentRoom();
        // Afficher la liste des objets (View)
        ListView<ObjetZork> view = View.buildMenu(room);
        view.show();
        // Afficher le menu principal (View)
        game.showMenu();
    }
}
GoCommand.java — Se déplacer vers une pièce voisine
public class GoCommand implements Command {
    private Gamer gamer;
    private ZorkMap map;

    @Override
    public void execute() {
        Room current = gamer.getCurrentRoom();
        // Récupérer les directions valides (Model)
        EnumMap<Direction, Room> voisines = map.nextValidLocation(current);
        // Afficher les directions et lire le choix (View)
        ListView<Direction> dirMenu = View.buildMenu(voisines.keySet());
        dirMenu.show();
        Direction choice = dirMenu.readChoice();
        // Déplacer le joueur (Model)
        gamer.moveTo(voisines.get(choice));
    }
}
À l'examen — comprendre le MVC La règle principale : le Model ne sait pas que la View existe. La View ne sait pas que le Model existe. Seul le Controller fait le lien. On te demandera souvent de classer des classes dans les bonnes couches, ou d'implémenter une Command.

Le pattern Builder

Création

Le problème : constructeurs à trop de paramètres

Imagine une classe UniteEnseignement avec 7 attributs. Son constructeur ressemblerait à :

new UniteEnseignement("POO2", "Design Patterns", 4, 1, 2, 1, 4);
// Que signifient 4, 1, 2, 1, 4 ? Impossible à lire !
Analogie — Commander un burger Au lieu de dire "Je veux un burger pain-brioche-steak-cheddar-salade-tomate-pickles-sauce", tu coches les options sur une fiche. Le Builder c'est cette fiche : tu remplis ce que tu veux, le reste prend les valeurs par défaut.

Structure du Builder

  1. Une classe Builder distincte avec les mêmes attributs que la classe cible
  2. Constructeur du Builder : uniquement les paramètres obligatoires
  3. Méthodes withXxx() pour les paramètres optionnels — renvoient this pour le chaînage
  4. Méthode build() qui crée l'objet final
  5. La classe cible a un constructeur package-private prenant le Builder
Builder — Exemple complet (UEBuilder)
// 1. La classe Builder
public class UEBuilder {
    String name;
    String description;
    int ects = 4;         // valeur par défaut
    int coefP1 = 1;       // valeur par défaut
    int coefP2 = 2;       // valeur par défaut
    int coefTP = 1;       // valeur par défaut
    int numSemestre = 1;  // valeur par défaut

    // Constructeur : uniquement les paramètres obligatoires
    public UEBuilder(String name, String description) {
        if (name == null || description == null) throw new NullPointerException();
        this.name = name;
        this.description = description;
    }

    // Méthodes optionnelles — renvoient this pour le chaînage
    public UEBuilder withEcts(int val)        { ects = val;        return this; }
    public UEBuilder withCoefP1(int val)     { coefP1 = val;     return this; }
    public UEBuilder withCoefP2(int val)     { coefP2 = val;     return this; }
    public UEBuilder withNumSemestre(int val){ numSemestre = val; return this; }

    // build() : crée l'objet final
    public UniteEnseignement build() {
        return new UniteEnseignement(this);
    }
}

// 2. La classe cible — constructeur package-private prend le Builder
public class UniteEnseignement {
    private final String name;
    private final int ects;
    private final int coefP1, coefP2, coefTP, numSemestre;

    // Package-private : seul UEBuilder.build() peut appeler ça
    UniteEnseignement(UEBuilder b) {
        name = b.name;
        ects = b.ects;
        coefP1 = b.coefP1;
        coefP2 = b.coefP2;
        coefTP = b.coefTP;
        numSemestre = b.numSemestre;
    }
}

// 3. Utilisation — lisible grâce au chaînage !
UniteEnseignement poo2 = new UEBuilder("POO2", "Design Patterns")
    .withEcts(4)
    .withCoefP2(2)
    .withNumSemestre(4)
    .build();  // ← crée l'objet final
À l'examen — les 3 points clés du Builder
1. Méthodes withXxx() retournent this (pour le chaînage)
2. Le constructeur de la classe cible est package-private (ni public ni private)
3. build() est la seule façon de créer l'objet

Quiz de révision

1. Quelle méthode de l'interface Iterator permet de savoir s'il reste des éléments à parcourir ?

hasNext() renvoie true s'il reste au moins un élément. next() renvoie l'élément et avance le curseur. size() n'existe pas dans Iterator. peek() n'existe pas dans Iterator.

2. Pour qu'une classe soit utilisable dans un for-each, elle doit implémenter :

Iterable<E> possède la méthode iterator() qui renvoie un Iterator. Java utilise cette méthode en coulisse pour le for-each. Un Iterator en lui-même ne suffit pas — c'est Iterable qui "ouvre la porte" au for-each.

3. Dans le pattern Singleton, pourquoi le constructeur est-il déclaré private ?

Le constructeur private empêche d'écrire "new MonSingleton()" depuis une autre classe. Seule la classe elle-même peut créer son instance (via la variable static finale theInstance).

4. Quel est l'avantage principal du pattern Strategy par rapport aux if/else imbriqués ?

Le pattern Strategy encapsule chaque algorithme dans une classe séparée. La classe principale reçoit la stratégie en paramètre, donc on peut la changer dynamiquement sans toucher au code de la classe — respect du principe "ouvert/fermé".

5. Dans le pattern Decorator (PizzaAvecOlives), quelle interface implémente le décorateur ?

C'est LA clé du Decorator : le décorateur implémente la MÊME interface que ce qu'il décore. Ça permet de les empiler : PizzaAvecOlives(PizzaAvecOignons(new SimplePizza())). Chaque couche peut être une pizza comme les autres.

6. Quelle est la différence entre une opération intermédiaire et une opération terminale sur un Stream ?

Les opérations intermédiaires (filter, map) sont "paresseuses" : elles ne font rien toutes seules, elles produisent un nouveau Stream configuré. Le calcul réel ne se déclenche que quand on appelle une opération terminale (toList, count, iterator...). Après ça, le Stream est consommé.

7. Dans l'architecture MVC, quelle couche NE DOIT PAS avoir d'affichage ni d'interaction utilisateur ?

Le Model contient uniquement les données et la logique métier (Piece, Gamer, ObjetZork...). Il ne doit JAMAIS faire de System.out.println() ni lire de saisie utilisateur. C'est la View qui gère tout ce qui est affiché, et le Controller qui fait le lien.

8. Dans le pattern Builder, que renvoient les méthodes withXxx() ?

Les méthodes withXxx() retournent this pour permettre le chaînage : new UEBuilder("POO2", "...").withEcts(4).withCoefP2(2).build(). C'est build() qui renvoie l'objet final. Si withXxx() retournait void, on ne pourrait pas chaîner.
